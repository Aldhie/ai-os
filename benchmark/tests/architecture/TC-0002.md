## Question

```
Design production inference infrastructure for a 70B parameter LLM that must:
- Serve 500 concurrent users
- P95 TTFT (Time to First Token) < 2 seconds
- P95 throughput: 30 tokens/second per user
- Support streaming responses
- 99.9% availability
- Monthly budget: $50,000

Assume: NVIDIA H100 80GB SXM5 at $3.50/hour per GPU on cloud.

Deliver:
1. GPU requirements (number of GPUs, parallelism strategy)
2. Serving framework recommendation with justification
3. Batching strategy (static vs continuous batching)
4. Monthly cost estimate
5. How to handle traffic spikes (2x peak for 15 minutes)
6. Monitoring strategy (which specific metrics to track)
```

---

## Expected Behaviour

1. **GPU requirements**: 70B model in FP16 = 140GB VRAM → minimum 2× H100 80GB per replica (tensor parallelism). For 500 concurrent users at 30 tok/s = 15,000 tok/s aggregate. Estimate 4-8 replicas → 8-16 H100s for throughput target.
2. **Serving framework**: vLLM (preferred for PagedAttention + continuous batching) or TensorRT-LLM + Triton (NVIDIA-native). Must justify.
3. **Batching**: Continuous batching (not static) — model must know this distinction and why it matters for P95 latency.
4. **Cost**: 8-16 H100s × $3.50/hr × 730hr/month = $20,440–$40,880/month. Within $50K budget with buffer.
5. **Traffic spikes**: Horizontal pod autoscaling with pre-warmed replicas or spot instance buffer pool.
6. **Monitoring**: TTFT, tokens/second, GPU utilization, KV cache hit rate, queue depth, OOM events.

## Evaluation Criteria

| Criterion | Weight | Description |
|-----------|--------|-------------|
| GPU count and parallelism | 25% | Correct VRAM math and parallelism strategy |
| Continuous batching justification | 25% | Must name PagedAttention or equivalent insight |
| Cost estimate accuracy | 20% | Within 2x of expected range |
| Spike handling strategy | 15% | Specific mechanism |
| Monitoring completeness | 15% | At least 4 relevant metrics named |

---

## Failure Conditions

- Claims a single H100 can serve a 70B model
- Recommends static batching without acknowledging latency cost
- Cost estimate >$200K/month
- Monitoring list is generic ("monitor CPU, memory")
