## Question

```dockerfile
# Current production Dockerfile — image is 2.1GB and takes 8 minutes to build
FROM python:3.11

WORKDIR /app

COPY . .

RUN apt-get update && apt-get install -y gcc git curl wget vim
RUN pip install -r requirements.txt
RUN pip install pytest coverage black flake8

ENV DEBUG=true
ENV SECRET_KEY=hardcoded-secret-key-do-not-share

CMD ["python", "app.py"]
```

Identify ALL problems and provide an optimized production Dockerfile.
```

---

## Expected Behaviour

Model identifies all of:
1. **`COPY . .` before `pip install`** — breaks layer cache on every code change. Fix: copy `requirements.txt` first, run pip install, then copy source.
2. **Dev tools in production** — `git curl wget vim pytest coverage black flake8` have no place in a production image. Each adds attack surface and size.
3. **Hardcoded `SECRET_KEY`** — secrets must NEVER be in Dockerfile (they appear in image layers and `docker history`). Fix: inject at runtime via environment variable.
4. **`DEBUG=true`** in production — this is a critical security and performance issue.
5. **No non-root user** — running as root is a security risk. Fix: `RUN adduser` + `USER`.
6. **`python:3.11` full image** — use `python:3.11-slim` as base for ~300MB savings.
7. **Multi-stage build not used** — build dependencies (gcc) can be in a builder stage, not final image.
8. **No `.dockerignore`** — `COPY . .` would copy `.git`, `__pycache__`, test files, etc.

## Evaluation Criteria

| Criterion | Weight | Description |
|-----------|--------|-------------|
| Layer cache fix | 20% | requirements.txt before source code |
| Hardcoded secret removal | 25% | Critical security issue |
| Non-root user | 20% | Security hardening |
| Dev tools removal / slim base | 20% | Size optimization |
| Multi-stage or correct fix provided | 15% | Actual corrected Dockerfile |

## Failure Conditions

- Keeps hardcoded SECRET_KEY in any form in the Dockerfile
- Does not address the layer cache ordering
- Keeps DEBUG=true
