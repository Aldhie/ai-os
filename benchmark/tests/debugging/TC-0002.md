## Question

```go
// Bug report: occasionally returns wrong count, sometimes panics.
// Runs under high concurrency (1000+ goroutines).

package main

import (
    "fmt"
    "sync"
)

type Counter struct {
    mu    sync.Mutex
    count int
    log   []int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    c.count++
    c.mu.Unlock()
    c.log = append(c.log, c.count) // BUG: outside lock
}

func (c *Counter) Reset() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.count = 0
    c.log = nil
}

func (c *Counter) GetLog() []int {
    return c.log // BUG: no lock, returns live slice
}

func main() {
    c := Counter{}
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            c.Increment()
        }()
    }
    wg.Wait()
    fmt.Println(c.GetLog())
}
```

Find all race conditions, explain exactly why each is a data race, and provide the corrected code.
```

---

## Expected Behaviour

1. **`c.log = append(c.log, c.count)` outside the mutex** — `c.count` is read after the mutex is released; another goroutine may have already incremented it. `append` on `c.log` is also unprotected → data race on slice header (pointer, len, cap).
2. **`GetLog()` returns live `c.log` slice without lock** — caller holds a reference to the underlying array while other goroutines may be `append`ing → concurrent read+write on slice backing array.
3. **`Reset()` sets `c.log = nil`** while a goroutine in `Increment()` may have just read `c.log` for append (read between the Lock in Increment and the append outside it).

Correct fix: move both `c.count++` and `append` inside the same lock acquisition. `GetLog()` must lock and return a copy.
